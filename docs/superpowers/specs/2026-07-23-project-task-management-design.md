# 项目任务管理设计

## 背景

项目成员目前缺少项目内的轻量任务协作能力。日常事项需要在项目之外记录，任务负责人、完成状态、截止时间和项目上下文无法集中维护。

本设计在项目详情中增加类似 Microsoft To Do 的快速任务管理能力，延续现有飞书项目风格，并遵循系统现有的项目成员和权限模型。

## 目标

- 项目成员可以在项目内快速创建、查看和完成任务。
- 创建任务时只要求填写标题，负责人默认当前用户，也可以立即改为其他项目成员。
- 截止时间选填，精确到小时，可在任务创建后或完成后补录。
- 支持任务详情、检查项、优先级、负责人、提醒及软删除。
- 任务可选关联项目阶段或里程碑，但不参与阶段门禁。
- 所有项目成员可查看任务，创建人、负责人和项目经理可协同维护。
- 提供站内到期提醒，并为未来增加邮件、飞书等渠道保留数据扩展能力。

## 非目标

- 不引入审批流、任务依赖、甘特图、工时、子任务树或复杂状态机。
- 不让任务状态影响项目阶段推进。
- 本期不发送邮件、短信或飞书通知。
- 不复用具有研发交付和成本归集含义的 `custom_dev_task`。
- 不新增前端第三方组件库。

## 方案选择

### 采用方案：独立项目任务模型

新增独立的项目任务、检查项和提醒数据。该方案可以保持领域含义清晰，并可靠支持多人权限、筛选、并发编辑和提醒。

### 未采用方案

- 复用 `custom_dev_task`：表面改动较少，但会把通用待办与研发交付、成本归集语义混在一起。
- 将任务以 JSON 附着到项目：初期代码少，但难以查询、并发编辑和生成提醒。

## 页面与交互

### 项目任务页签

项目详情增加“项目任务”页签，采用列表加详情面板的三栏结构：

- 左侧筛选：我的任务、全部任务、今天、已逾期、已完成。
- 中间列表：按截止日期分组展示任务。
- 右侧详情：展示和编辑当前选中的任务。

页面随任务数量自然向下延伸，不使用固定高度的区域滚动。右侧详情可保持吸顶，但其内容仍随页面滚动。

列表和卡片延续现有飞书项目风格，统一间距、高度、圆角、边框和状态颜色。所有按钮、状态和空状态文案使用中文。

### 快速创建

中间列表顶部提供快速创建行：

- 标题为唯一必填项。
- 负责人默认当前用户。
- 可以在创建前直接改选其他当前项目成员。
- 截止时间选填，使用精确到小时的日期时间选择。
- 按回车或点击“创建任务”完成创建。

创建成功后任务立即出现在当前列表，并自动选中以便继续补充详情。创建失败时保留输入内容，并展示明确的中文错误信息。

### 任务列表

每条任务统一展示：

- 完成复选框；
- 任务标题；
- 负责人头像或姓名；
- 截止时间；
- 优先级标识；
- 检查项完成进度；
- 可选的阶段或里程碑标签。

没有截止时间时显示“未设置截止时间”，并归入“无截止时间”分组。

“今天”仅包含截止小时位于当前自然日且未完成的任务。“已逾期”仅包含有截止时间、截止小时早于当前小时且未完成的任务。

点击完成复选框将状态切换为“已完成”，再次点击可以重新打开。完成任务不会锁定详情，仍可补录截止时间、描述和检查项。

### 任务详情

右侧详情支持编辑：

- 标题；
- 状态；
- 负责人；
- 截止时间；
- 优先级；
- 描述；
- 检查项；
- 提醒时间；
- 项目阶段；
- 项目里程碑。

负责人只能选择当前项目成员。阶段和里程碑只能选择当前项目下的有效记录，均为选填。

删除操作放在次要操作区，必须二次确认。删除采用软删除，不在正常列表中展示。

## 任务生命周期

任务只有两个业务状态：

- `TODO`：待完成；
- `DONE`：已完成。

完成时记录完成人和完成时间，重新打开时清空这两个字段。软删除独立于业务状态，不新增“已删除”状态。

任务创建、详情补充、完成、重新打开和软删除都记录现有审计字段。任务不因项目阶段推进自动改变状态。

## 数据设计

### `project_task`

主要字段：

- `id`：自增主键；
- `org_id`、`project_id`；
- `title`、`description`；
- `status`：`TODO` 或 `DONE`；
- `priority`：低、普通、高，默认普通；
- `creator_user_id`、`assignee_user_id`；
- `due_at`：可空，按小时保存；
- `stage_code`：可空；
- `milestone_id`：可空；
- `completed_by_user_id`、`completed_at`；
- `version`：乐观锁版本；
- `deleted`、`deleted_at`、`deleted_by_user_id`；
- `created_at`、`updated_at`。

`project_id + status + deleted`、`assignee_user_id + status + deleted` 和 `due_at + status + deleted` 建立查询索引。

### `project_task_check_item`

主要字段：

- `id`：自增主键；
- `task_id`；
- `content`；
- `completed`；
- `sort_order`；
- `created_at`、`updated_at`。

检查项随任务详情一起查询和保存，不设计多级嵌套。

### `project_task_reminder`

主要字段：

- `id`：自增主键；
- `task_id`、`recipient_user_id`；
- `channel`：本期只写入 `IN_APP`；
- `remind_at`；
- `read_at`；
- `created_at`、`updated_at`。

任务有截止时间时，默认提醒时间为截止前 30 分钟，用户可修改或关闭。没有截止时间时不创建提醒；后续补录截止时间时再生成默认提醒。已完成任务补录截止时间时默认不自动生成提醒，只有用户主动设置提醒才创建。

负责人变化时，未读站内提醒同步变更接收人。任务完成、重新打开、删除或修改截止时间时，对未读提醒进行相应取消、恢复或重算。

未来增加外部通知时，可以为同一任务增加其他 `channel` 的提醒记录，无需修改任务主表；本期不实现外部发送器。

## 接口设计

在项目接口下增加任务资源：

- `GET /api/v1/projects/{projectId}/tasks`：按负责人、状态、截止范围等条件查询；
- `POST /api/v1/projects/{projectId}/tasks`：快速创建任务；
- `GET /api/v1/projects/{projectId}/tasks/{taskId}`：查询任务详情；
- `PUT /api/v1/projects/{projectId}/tasks/{taskId}`：保存任务详情及检查项；
- `POST /api/v1/projects/{projectId}/tasks/{taskId}/complete`：完成任务；
- `POST /api/v1/projects/{projectId}/tasks/{taskId}/reopen`：重新打开；
- `DELETE /api/v1/projects/{projectId}/tasks/{taskId}`：软删除任务。

增加当前用户的站内提醒接口：

- `GET /api/v1/task-reminders/unread`：查询已经到达提醒时间的未读提醒；
- `POST /api/v1/task-reminders/{reminderId}/read`：标记已读。

任务列表返回列表展示所需的检查项计数和负责人姓名，避免逐条补查。任务详情返回完整检查项。

## 权限规则

- 当前项目成员以及现有跨项目管理角色可以查看任务。
- 当前项目成员可以创建任务。
- 创建人、当前负责人、项目经理及现有具备跨项目写权限的管理角色可以编辑、完成和重新打开任务。
- 创建人、项目经理及现有具备跨项目写权限的管理角色可以软删除任务。
- 负责人必须是当前项目的有效成员。
- 普通用户只能读取和标记属于自己的提醒。

所有权限必须在后端校验，前端只负责隐藏或禁用无权操作。

项目成员退出后，已有任务仍保留其历史负责人。该成员不再出现在负责人候选列表中，项目经理可以将任务重新分配给当前成员。

## 并发、校验与异常处理

- 标题去除首尾空格后不能为空，并限制合理长度。
- 截止时间可空；非空值统一截断到小时后保存和比较。
- 阶段、里程碑和负责人必须属于当前项目。
- 更新请求携带 `version`，版本冲突时提示“任务已被其他成员更新，请刷新后重试”。
- 列表请求失败时保留页面结构并提供“重新加载”。
- 创建和编辑失败时保留用户输入，不静默丢失内容。
- 已软删除任务不能继续编辑、完成或重新打开。

## 站内提醒

系统顶部增加轻量提醒入口，前端每分钟查询一次到达提醒时间的未读提醒。提醒条目展示项目名称、任务标题和截止时间，点击后进入对应项目的“项目任务”页签并选中任务。

提醒查询由数据库时间条件直接筛选，不引入新的消息中间件或定时任务。读取提醒和打开任务不会自动标记已读，用户可以明确执行“标记已读”。

## 前端结构

任务功能作为项目模块内的独立组件接入现有 `ProjectDetail` 页签，不将全部逻辑继续堆入项目详情文件。

复用项目查询、当前登录用户、项目成员选择、里程碑选择和现有请求封装。状态更新后只刷新任务和提醒相关缓存，不重新加载整个项目详情。

快速创建和完成操作可以进行即时界面反馈；请求失败时回滚并展示错误。详情保存以服务端返回的新版本为准。

## 测试范围

### 后端

- 项目成员创建、查看和筛选任务。
- 非项目成员被拒绝访问。
- 创建人、负责人、项目经理和跨项目管理角色的编辑权限。
- 普通成员不能编辑或删除无关任务。
- 负责人、阶段和里程碑的项目归属校验。
- 截止时间为空、按小时保存以及完成后补录。
- 完成、重新打开和软删除的状态变化。
- 乐观锁冲突不会覆盖新数据。
- 无截止时间不生成提醒，补录截止时间后生成提醒。
- 完成任务补录截止时间时不会自动生成提醒。
- 提醒只对接收人可见并可标记已读。

### 前端

- 项目详情显示“项目任务”页签。
- 快速创建仅填写标题即可成功，负责人默认当前用户。
- 创建时可以修改负责人和选填小时级截止时间。
- 筛选、日期分组、无截止时间和逾期状态正确。
- 详情可以编辑、保存和补录截止时间。
- 完成任务可重新打开，完成后仍可编辑详情。
- 无权限操作正确隐藏或禁用。
- 所有操作按钮和状态文案为中文。
- 任务列表随页面增长，不出现固定区域滚动。
- 站内提醒入口能打开对应项目任务。

## 验收标准

1. 项目成员进入项目后，可以在“项目任务”页签仅输入标题快速创建任务。
2. 创建时负责人默认自己，也可以选择其他项目成员。
3. 截止时间不是必填项，精确到小时，可在创建后或任务完成后补录。
4. 任务可以查看详情、维护检查项、完成、重新打开和软删除。
5. 列表支持我的任务、全部任务、今天、已逾期和已完成筛选。
6. 任务可选关联当前项目阶段或里程碑，但不影响阶段推进。
7. 有截止时间的未完成任务按配置产生站内提醒，无截止时间任务不提醒。
8. 权限、并发和项目数据归属由后端可靠校验。
9. 页面样式与现有系统一致，中文交互完整，数据较多时页面整体延伸。
