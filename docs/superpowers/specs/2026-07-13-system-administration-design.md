# 系统管理模块补全设计

## 1. 目标

把当前仅有占位页、无点击行为的“系统管理”补成系统管理员可实际使用的管理台，覆盖用户与团队、角色权限、产品与版本、审计日志、系统配置五项能力。所有写操作受 `system:manage` 权限和 CSRF 保护，并写入审计日志。

## 2. 范围与原则

- 复用现有 React、Ant Design、TanStack Query、Spring MVC、JdbcTemplate 和既有数据表，不引入新依赖。
- 使用状态切换代替物理删除，避免破坏项目、文件、审计等外键关系。
- 管理数据按当前登录用户的 `organizationId` 隔离；请求体不能指定其他组织。
- 内置六类角色的编码与名称不可修改，只允许管理员修改权限集合。
- 产品目录沿用现有全局产品与版本模型；产品编码和版本名创建后不可修改。
- 敏感密钥、数据库连接和对象存储凭据继续由环境变量维护，不进入系统配置表。

## 3. 前端信息架构

`/admin` 重定向到 `/admin/users`。系统管理的二级导航使用真实链接，并根据当前路径高亮：

| 路由 | 页面 | 核心操作 |
| --- | --- | --- |
| `/admin/users` | 用户与团队 | 查看用户/团队；新增、编辑用户；分配主团队与角色；启用/停用；新增、编辑团队 |
| `/admin/roles` | 角色权限 | 查看六类角色；按模块勾选权限；保存权限集合 |
| `/admin/products` | 产品与版本 | 查看、新增、编辑产品；查看、新增、编辑版本；切换状态 |
| `/admin/audit-logs` | 审计日志 | 按关键字、动作、资源类型、起止时间筛选；分页查看详情与 traceId |
| `/admin/settings` | 系统配置 | 编辑白名单配置并保存 |

页面统一采用“标题和主操作 + 筛选区 + 表格 + 抽屉表单”的现有交互模式。加载、空数据、接口失败和无权限均有明确状态；保存成功后关闭抽屉、提示成功并刷新关联查询。

## 4. 管理能力

### 4.1 用户与团队

用户列表返回主团队和角色，不返回密码摘要。新增用户必须填写用户名、至少 8 位初始密码、显示名，可选邮箱、主团队和角色。编辑时用户名不可变；可修改显示名、邮箱、主团队、角色与状态。停用用户后不能登录。

团队可新增和编辑名称、编码、上级团队与启用状态。团队不物理删除；团队编码在组织内唯一。用户和团队查询只返回当前管理员所在组织的数据。

### 4.2 角色权限

权限目录由后端返回 `code`、`name`、`module`。前端按模块分组显示复选框。保存时以完整集合替换目标角色权限，并在同一事务中记录审计。系统管理员角色不能移除 `system:manage`，避免平台失去管理入口。

### 4.3 产品与版本

复用现有产品目录服务。产品支持新增以及修改名称、负责人、分类、状态；版本支持新增以及修改发布日期和状态。所有产品写接口增加 `system:manage` 鉴权，读接口保持登录用户可读，以免影响项目和其他业务模块。

### 4.4 审计日志

管理端审计查询以当前组织为边界，按创建时间倒序分页。筛选项包括自由关键字（操作者、动作、资源、traceId、详情）、动作、资源类型、开始时间和结束时间。响应返回 `items`、`page`、`pageSize`、`total`，默认每页 20 条，最大 100 条。审计日志只读。

### 4.5 系统配置

系统配置不是任意键值编辑器，只暴露以下白名单：

| 键 | 类型与约束 | 默认值 | 作用 |
| --- | --- | --- | --- |
| `platform.name` | 1–40 字符 | `智鹿交付` | 侧栏品牌名称 |
| `platform.environmentLabel` | 1–20 字符 | `内部生产环境` | 顶栏环境标识 |
| `platform.timezone` | IANA 时区 | `Asia/Shanghai` | 审计时间展示 |
| `platform.supportEmail` | 合法邮箱或空 | 空 | 管理台支持联系方式 |
| `agent.timeoutMinutes` | 1–240 整数 | 环境配置 `delivery.agent.timeout-minutes`，默认 30 | 新建 Agent 任务超时时间 |

`GET /api/v1/admin/settings` 返回合并默认值后的结构；`PUT` 一次保存全部白名单值，拒绝未知键和非法值。平台名称与环境标识保存后在应用壳层刷新；Agent 超时仅影响之后新建的任务。

## 5. 后端接口

在 `/api/v1/admin` 下补齐：

- `GET/POST /users`、`PUT /users/{id}`、`PUT /users/{id}/status`
- `GET/POST /teams`、`PUT /teams/{id}`
- `GET /roles`、`GET /permissions`、`PUT /roles/{id}/permissions`
- `GET /audit-logs`
- `GET/PUT /settings`

产品与版本继续使用 `/api/v1/products`。服务层负责组织隔离、唯一性冲突、状态/权限白名单和不存在数据的稳定错误；控制器只做请求映射、当前用户传递和审计上下文组装。

## 6. 数据流与错误处理

1. `RequirePermission` 阻止无 `system:manage` 的用户进入管理路由。
2. 页面通过管理 API 读取数据，TanStack Query 维护缓存和加载状态。
3. 表单提交携带 CSRF 头；后端从认证主体取得组织与操作者，不信任前端组织字段。
4. 服务在事务中校验并写业务数据；控制器/服务记录成功写操作的操作者、资源和摘要。
5. 前端把 `ApiError.message` 展示为操作失败提示；校验冲突保持 400/409，越权保持 403，不存在保持 404。

为避免管理员自锁，后端拒绝两类操作：停用当前登录用户，以及从 `ADMIN` 角色移除 `system:manage`。

## 7. 测试与验收

后端集成测试覆盖：组织隔离、用户编辑与角色分配、团队编辑、权限目录与防自锁、审计筛选分页、配置白名单与 Agent 动态超时、产品写权限。前端组件测试覆盖：五个二级菜单可导航并高亮、各页面真实加载、关键新增/编辑/保存流程、接口错误提示。

最终验证命令：

```bash
cd backend && mvn test
cd frontend && pnpm test -- --run && pnpm build
```

浏览器验收使用管理员账号依次进入五个页面，各完成一次新增或保存，刷新后数据仍存在；再用非管理员账号确认系统管理入口不可见且 API 返回 403。
