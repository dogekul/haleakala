# 客户中心看板布局优化设计

## 背景

客户中心的“售前推进”和“客户运营”页面目前存在以下体验问题：

- 看板留白较多，信息层级偏弱，同屏可浏览的数据量较少。
- 卡片内容和操作按钮数量不同，造成卡片高度与操作区位置不一致。
- 售前推进页仍有 `PASS`、`REJECT` 英文按钮，与系统中文界面不一致。
- 客户运营页的筛选区占用空间较大，三列卡片的视觉关系不够紧凑。

本次选择“紧凑型高密度看板”方向，在不改变业务流程和接口的前提下统一两个页面的布局、卡片和按钮样式。

## 目标

- 提升售前推进和客户运营页面的信息密度与浏览效率。
- 让同一看板内的所有卡片尺寸、信息层级和操作区位置保持一致。
- 将用户可见的英文操作按钮替换为中文。
- 复用现有 React、Ant Design 组件和全局样式体系，不增加新依赖。

## 页面结构

### 售前推进

- 保留“线索、立项评估、POC 验证、投标、合同”五阶段横向看板。
- 五列使用统一列宽和间距，列头统一展示“阶段名称 + 数量徽标”。
- 减少列容器内边距和卡片间距，提高同屏信息量。
- 卡片随数据自然向下排列并拉长整个页面，不设置看板内部纵向滚动。
- 屏幕宽度不足以容纳五列时，仅允许看板整体横向滚动，避免阶段列被过度压缩。

### 客户运营

- 保留“回款/维保、持续运营、复购”三列看板。
- 搜索框和负责人、客户、阶段、状态筛选项收敛为单行工具栏，开放/关闭数量放在工具栏右侧。
- 三列使用与售前推进一致的列头、数量徽标、卡片和操作区规范。
- 卡片随数据自然向下排列并拉长整个页面，不设置看板内部纵向滚动。
- 窄屏下三列依次降为两列和单列，避免卡片内容过窄。

## 卡片规范

- 同一页面内所有卡片使用统一固定高度、内边距、圆角和边框。
- 卡片主体采用纵向弹性布局，操作区固定贴在卡片底部；按钮数量不同不影响卡片高度。
- 标题最多显示一行，超出部分省略，悬停时通过原生标题提示显示完整文本。
- 客户、负责人等辅助信息压缩为一行，并使用较弱的文字颜色与标题形成层级。
- 普通操作使用白色描边按钮，主要推进操作使用蓝色按钮，拒绝、关闭等终止操作使用浅红色危险按钮。
- 操作按钮保持一致高度，紧凑但保留可点击面积。

## 中文按钮

售前推进页的英文按钮按以下规则替换：

- `PASS` → `通过`
- `REJECT` → `拒绝`

其他现有按钮统一使用明确中文：

- `阶段文档` → `查看文档`
- `产出物` → `查看产出物`
- `推进` → `推进阶段`
- `转交实施` 保持不变
- 客户运营中的 `编辑` 保持不变
- 客户运营中的推进按钮显示为 `推进阶段`
- 客户运营的终止动作显示为 `关闭运营`

按钮只调整显示文案和视觉样式，不改变原有事件处理、权限和阶段流转逻辑。

## 响应式与滚动

- 售前推进以五列桌面看板为主；宽度不足时使用页面内的横向看板滚动。
- 客户运营在桌面端为三列，在中等宽度下降为两列，在较窄宽度下降为单列。
- 两个页面均不使用看板内部纵向滚动；卡片数量增加时自然拉长页面。
- 页面顶部标题、说明、主按钮和客户中心二级导航保持现有结构。

## 实现范围

主要修改以下前端文件：

- `frontend/src/modules/customer-center/PresaleBoardPage.tsx`
- `frontend/src/modules/customer-center/OperationBoardPage.tsx`
- `frontend/src/styles/global.css`
- 对应的客户中心页面测试文件

实现优先通过现有 className 和组合 CSS 选择器完成，不引入新的通用组件或第三方依赖。仅在避免重复结构确有必要时提取很小的展示组件。

## 测试与验收

- 更新页面测试，验证“通过”“拒绝”“查看文档”“查看产出物”“推进阶段”“关闭运营”等中文文案。
- 验证现有推进、通过、拒绝、转交实施、编辑和关闭操作仍调用原有处理逻辑。
- 验证售前推进和客户运营的页面加载、筛选及数据展示不受影响。
- 运行相关前端单元测试和构建检查。
- 在桌面宽度和窄屏宽度下人工检查卡片等高、按钮贴底、自然页面增长和响应式布局。

## 非目标

- 不修改后端接口、数据库结构和业务状态机。
- 不新增拖拽排序、批量操作或新的商机/运营字段。
- 不改变客户中心其他页面的功能和布局。
