# 客户管理模块设计

## 目标

新增独立的客户主数据模块，用于维护客户基本信息，并让创建项目时从启用客户中选择。客户不使用人工编码，数据库自增 `id` 直接作为主键和关联标识。

## 范围

首期包含客户查询、新建、编辑、启用/停用、列表/卡片展示，以及项目创建时的客户选择。不包含客户删除、行业、商机、合同、跟进记录、联系人子表或客户编码规则。

客户字段如下：

- 客户名称：必填，同一组织内唯一。
- 客户简称：可选。
- 联系人：可选。
- 联系电话：可选。
- 邮箱：可选，填写时校验格式。
- 地址：可选。
- 状态：`ACTIVE` 或 `INACTIVE`，新建默认启用。
- 备注：可选。

## 数据设计

新增 Flyway `V12` 迁移和 `customer` 表：

- `id BIGINT AUTO_INCREMENT PRIMARY KEY`
- `organization_id BIGINT NOT NULL`
- `name VARCHAR(180) NOT NULL`
- `short_name VARCHAR(100) NULL`
- `contact_name VARCHAR(100) NULL`
- `phone VARCHAR(40) NULL`
- `email VARCHAR(160) NULL`
- `address VARCHAR(500) NULL`
- `status VARCHAR(24) NOT NULL DEFAULT 'ACTIVE'`
- `remark TEXT NULL`
- `version BIGINT NOT NULL DEFAULT 0`
- `created_at`、`updated_at`

`customer` 通过外键归属组织，并以 `(organization_id, name)` 唯一约束防止同组织重名。表中不创建 `code` 字段。

`delivery_project` 新增可空 `customer_id` 外键，并保留现有 `customer_name`。迁移按“组织 + 非空客户名称”创建基础客户记录，再回填历史项目的 `customer_id`。`customer_name` 继续作为兼容快照，读取项目时优先使用关联客户的当前名称，无关联时回退快照。

## 后端设计

新增独立 `customer` 包，沿用现有 Spring Boot、JdbcTemplate 和 Map/DTO 风格：

- `GET /api/v1/customers`：按当前组织查询，支持 `keyword` 和 `status`。
- `POST /api/v1/customers`：创建客户，数据库生成 `id`。
- `PUT /api/v1/customers/{id}`：按 `version` 乐观锁更新客户。

不提供删除接口。停用客户保留历史关联，但不能用于新项目。

项目创建请求由 `customerName` 改为 `customerId`。服务端必须验证客户存在、属于项目组织且为 `ACTIVE`，然后保存 `customer_id` 和当前客户名称快照。项目响应增加 `customerId`，保留 `customerName`，避免破坏现有驾驶舱和项目视图。

新增 `customer:read` 与 `customer:write` 权限：

- ADMIN：读写。
- PMO：读写。
- DELIVERY_MANAGER：读写。
- DELIVERY_ENGINEER、TECH_MANAGER、PRODUCT_MANAGER：只读。

所有查询和写入按当前用户组织隔离。重复名称和乐观锁冲突返回 `409`，越权或不存在返回 `404`，无权限返回 `403`，停用客户用于新项目返回明确的 `400`。创建、编辑和状态变更写入审计日志。

## 前端设计

主导航在“驾驶舱”和“项目空间”之间增加“客户管理”，使用独立 `customer:read` 控制可见性。

客户页面延续现有飞书项目风格：

- 默认高密度列表，保留卡片视图切换。
- 页面标题区显示客户总数、启用数和停用数。
- 工具栏提供名称、简称或联系人搜索，以及状态筛选。
- 列表展示客户名称、简称、联系人、电话/邮箱、状态、关联项目数和更新时间。
- 卡片视图突出客户名称、联系信息、状态和项目数量，长文本截断，避免溢出。
- 新建和编辑使用右侧抽屉，两列紧凑表单；名称单独占一行，地址和备注使用多行输入。
- 页面不展示客户编码；需要定位记录时可弱化显示 `ID: <id>`。
- 只读用户不显示新建、编辑和状态切换控件。

“项目空间”的新建项目抽屉和“驾驶舱”的快速创建抽屉都把客户名称输入框替换为可搜索 `Select`。选择器只加载当前组织的启用客户，以名称为主、简称为辅展示；没有启用客户时展示清晰空状态，有写权限的用户可跳转客户管理创建客户。

## 数据流

1. 用户进入客户管理页面，前端按权限加载本组织客户。
2. 新建或编辑通过客户接口保存，成功后刷新客户查询缓存。
3. 创建项目时前端提交 `customerId`。
4. 后端重新校验客户组织和状态，不信任前端选项。
5. 后端保存关联和名称快照，项目列表与详情继续通过 `customerName` 展示。

## 测试与验收

后端测试覆盖：

- V12 在现有迁移序列之后执行，创建客户表并回填历史项目。
- 客户 `id` 由数据库生成，请求中无客户编码。
- 客户增改查、同组织重名、乐观锁和状态校验。
- 跨组织客户不可查询、编辑或用于创建项目。
- 停用客户不能创建新项目，历史项目仍可读取。

前端测试覆盖：

- 客户列表、卡片切换、搜索、状态筛选和长文本不溢出。
- 新建/编辑抽屉不包含客户编码或行业字段。
- 只读权限隐藏写操作。
- 两个项目创建入口都提交 `customerId`，不再提交自由文本客户名称。

端到端验收流程：创建客户，编辑联系信息，使用该客户创建项目，确认项目列表和详情显示正确客户，再停用客户并确认其不再出现在新建项目选择器中。
