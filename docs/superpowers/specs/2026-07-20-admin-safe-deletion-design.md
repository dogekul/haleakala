# 系统管理安全删除设计

## 目标

为系统管理中的用户、团队和角色提供可见、可审计的删除能力，同时保护当前登录账号、内置角色和已经产生业务历史的数据。

## 当前问题

- 用户与团队页面只有新增、编辑、启停操作，没有删除入口。
- 角色权限页面只有权限配置，没有删除入口。
- 后端没有用户、团队或角色的 `DELETE` 接口。
- 用户被项目、商机、知识、文件、审计等业务表广泛引用，直接级联删除会破坏历史记录。

## 删除规则

### 用户

- 当前登录用户不能删除。
- 删除用户时一并移除仅用于身份配置的角色关系、团队关系和 SSO 身份。
- 如果用户仍被交付、客户、产品、资源、知识、文件或审计等业务记录引用，则拒绝删除，并提示先停用用户。
- 删除成功后刷新用户列表并记录审计事件。

### 团队

- 有直接成员、附加团队成员关系或子团队时不能删除。
- 无任何引用的团队可直接删除。
- 删除成功后刷新团队列表并记录审计事件。

### 角色

- 内置角色不能删除，并在界面上明确显示保护状态。
- 已分配给用户的非内置角色不能删除。
- 删除非内置角色时先删除其权限关系，再删除角色本身。
- 删除成功后刷新角色列表并记录审计事件。

## API 与界面

- `DELETE /api/v1/admin/users/{id}`
- `DELETE /api/v1/admin/teams/{id}`
- `DELETE /api/v1/admin/roles/{id}`
- 成功统一返回 `204 No Content`。
- 列表和角色卡片使用中文危险操作按钮与二次确认。
- 后端冲突原因直接通过现有错误处理显示给用户。

## 实现约束

- 复用现有 Spring 事务、`ConflictException`、审计服务、React Query 和 Ant Design `Popconfirm`。
- 不新增依赖，不修改数据库结构，不级联删除业务历史。
- 数据库约束作为最终保护；服务层提供当前用户、成员、子团队、内置角色等明确校验。

## 验证

- 后端测试覆盖成功删除及所有保护条件。
- 前端测试覆盖 API 请求、确认入口、刷新列表和错误提示。
- 完整前后端测试与构建通过。
- 本地容器重建后，验证三个页面的按钮、确认框、成功刷新与受保护提示。
