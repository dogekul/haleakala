# 消保合规真实功能 Spec 生成设计

## 1. 背景与目标

产品 `XBHG / 消保合规` 当前包含 10 个一级能力域、31 个二级模块和 124 个原子功能，功能均为 `ACTIVE`，但 124 个功能的 `outline_link_id` 全部为空。知识库中已发布并启用“产品功能设计 Spec”模板，Outline 组织配置也已就绪，但模板正文仍是待填写骨架。

本次目标是在 Outline 的产品资料目录中，为 124 个原子功能分别创建一份可直接用于产品、研发、测试和合规评审的真实设计 Spec，并保证系统产品页能通过现有文档关联读取、编辑和导出这些文档。

## 2. 方案选择

采用“系统关联优先、限速初始化、领域化写入、闭环校验”的方案：

1. 通过现有产品文档业务键和确定性文档 ID 创建或恢复产品、模块与功能文档。
2. 对 Outline 请求限速并对 429 做退避重试，避免一次性创建 124 份文档触发服务限流。
3. 通过系统 Spec 读写接口取得最新 revision，再写入生成的 Markdown，保留乐观锁与数据库关联。
4. 完成后同时校验 MySQL 的 124 个关联、Outline 同步状态、文档标题、正文长度和占位符清理。

不采用直接绕过系统写 Outline 的方式，因为那会造成 Outline 文档与 `product_feature`、`outline_document_link` 脱节；不采用单纯复制模板，因为无法形成真实的功能设计输入。

## 3. 文档信息架构

每份 Spec 使用相同的信息架构，但内容由功能编码、所属能力域、二级模块和功能语义驱动：

1. 基本信息：产品、能力域、模块、功能编码、状态、文档版本。
2. 背景与业务目标：说明该功能解决的消保合规问题和可度量结果。
3. 范围与非目标：明确本功能负责和不负责的边界。
4. 用户角色与权限：业务提交人、合规人员、专家、管理员或审计人员的权限差异。
5. 前置条件与触发：数据、配置、权限和流程条件。
6. 主流程：可执行的用户步骤与系统行为。
7. 异常与补偿：缺失数据、重复提交、解析失败、规则冲突、超时或权限不足等处理。
8. 页面与交互：页面入口、字段、状态、按钮和中文反馈。
9. 业务规则：至少 5 条带编号的功能专属规则。
10. 数据设计：核心对象、关键字段、唯一性、状态和保留策略。
11. 接口与事件：请求、响应、幂等键、领域事件和失败语义。
12. 安全与审计：最小权限、数据范围、敏感信息、操作留痕。
13. 非功能指标：性能、容量、可用性和可观测性目标。
14. 验收标准：至少 6 条 Given/When/Then 风格可测试条件。
15. 依赖与后续：关联功能、外部系统和不阻塞上线的演进项。

正文不允许出现 `TODO`、`TBD`、空表格、反斜杠占位符或“请补充”等模板残留。

## 4. 领域化生成规则

生成器按二级模块定义业务对象、参与角色、状态语义、审计事件和性能基线，再按原子功能编码定义功能目标、核心动作、专属规则和验收场景。例如：

- `LAW-INGEST` 必须覆盖法规来源、发文机关、文号、发布日期、效力状态、原文校验和重复识别。
- `RISK-SEMANTIC` 必须覆盖模型置信度、阈值、证据片段、规则兜底、人工复核和模型版本追踪。
- `PRODUCT-FAIRNESS` 必须覆盖格式条款、公平性维度、权利义务失衡、免责/限责条款和人工定性。
- `DATA-RETENTION` 必须覆盖数据分类、保留期限、法律冻结、销毁证明和不可恢复删除。
- `OPS-MODEL-EVAL` 必须覆盖评测集版本、准确率/召回率、关键风险漏检、基线比较和发布门禁。

通用章节只负责格式与横向一致性，功能目标、规则和验收条件必须来自功能专属定义。

## 5. 写入流程

生成脚本使用本地后端 `http://localhost:8082`：

1. 以演示管理员登录并取得会话与 CSRF Token。
2. 查询产品、模块和功能，确认产品编码为 `XBHG` 且功能数为 124。
3. 分批触发产品文档同步；每批后查询文档树和数据库关联，只重试未关联功能。
4. 对每个功能读取当前 Spec 和 revision，生成 Markdown 后 PUT 保存。
5. 保存操作串行执行并在请求之间留出间隔；429、502、503 使用有上限的指数退避。
6. 生成过程支持重复执行：正文完全一致时跳过更新，已创建的确定性 Outline 文档不重复创建。

脚本不读取或输出 Outline Token，不直接修改 Outline 关联表，不删除已有文档。

## 6. 校验与完成标准

完成需同时满足：

- `product_feature.product_id=102` 共 124 条，且 124 条均有 `outline_link_id`。
- 每个关联的 `outline_document_link` 都有 `outline_document_id`，最终 `sync_status='READY'`。
- 124 个文档标题均为“功能名称 · 设计 Spec”。
- 每份正文不少于 1,200 个中文字符或等价 Markdown 长度，并包含全部 15 个章节。
- 正文不含 `{{...}}`、`TODO`、`TBD`、“请补充”或空业务规则。
- 随机抽检法规、智能审查、人工复核、消保专项、集成、安全、运营七个领域的 Spec，内容必须与功能语义一致。
- 产品页能够读取任一功能 Spec，并返回 Outline URL 和最新 revision。

## 7. 风险与控制

- Outline 429：串行请求、固定间隔与指数退避；只重试未完成项。
- 运行中断：使用确定性业务键和文档 ID，重跑不会产生重复文档。
- revision 冲突：每次保存前重新读取；冲突时重新拉取后只重试一次，不覆盖人工新改内容。
- 目录历史遗留：本次只创建当前 41 个能力模块和 124 个功能 Spec，不删除旧目录；目录清理由独立迁移处理。
- 生成质量：用功能专属规则表、占位符扫描和七领域抽检阻止泛化空文档进入完成状态。
