# 需求智能报告、编辑与废弃设计

## 目标

需求工坊完成采集时，不再机械替换模版字段，而是调用组织已配置的大模型，依据已发布的“需求调研报告模版”和真实业务上下文生成完整 Markdown 报告并存入 Outline。采集后的需求可编辑，也可作为保留历史记录的终态进行废弃。

## 范围

- 新建需求同步调用大模型生成调研报告。
- 编辑需求支持仅保存结构化字段，或明确选择覆盖并重新生成报告。
- 需求可废弃，废弃后保留编号、Outline 文档和审计记录。
- 列表和看板提供中文状态、编辑和废弃交互。
- 不增加异步队列、生成任务表、自动重试或报告版本表。

## 智能报告生成

`RequirementDocumentService` 继续负责模版选择、项目目录和 Outline 关联，并新增大模型生成职责。输入包含：

- 当前发布且启用的唯一需求调研报告模版标题、版本和 Markdown 快照；
- 需求编号、标题、描述、来源、优先级和采集人；
- 项目编号、项目名称、客户名称、产品和版本信息；
- 当前日期。

大模型必须返回严格 JSON，仅包含 `title` 和 `markdown`。提示词要求保留并补全模版结构，至少覆盖业务背景、相关角色、现状流程、问题痛点、建设目标、范围边界、详细需求、数据与接口、非功能要求、安全合规、验收标准、风险依赖和待确认事项。模型不得虚构事实；缺失内容统一标记为“待确认”。服务拒绝空标题、空正文和仍含 `{{...}}` 占位符的响应。

新建流程为：写入需求草稿 → 调用 AI → 建立项目需求目录 → 写入 Outline → 回写文档与模版关联。上述步骤位于同一数据库事务；AI、模版或 Outline 任一步失败时，需求记录回滚，前端保留抽屉内容并展示现有标准错误信息。

## 编辑与报告重生成

现有 `PUT /api/v1/requirements/{id}` 请求增加 `regenerateReport` 布尔值：

- `false`：只更新需求标题、描述、来源和优先级，不覆盖人工在 Outline 中编辑的内容；
- `true`：更新字段后，读取 Outline 最新修订，重新调用 AI，并以乐观锁覆盖报告；历史需求尚无文档时则创建并关联文档。

前端复用采集抽屉作为编辑抽屉。项目在编辑时只读，避免跨项目移动文档。抽屉提供“仅保存需求”和“保存并重新生成报告”两个动作；重新生成动作明确提示会覆盖 Outline 当前正文。`MERGED` 和 `ABANDONED` 状态不可编辑。

## 废弃

新增 `POST /api/v1/requirements/{id}/abandon`，请求只携带乐观锁 `version`。后端把非 `MERGED`、非 `ABANDONED` 的需求更新为 `ABANDONED`，递增版本并写入 `REQUIREMENT_ABANDONED` 审计事件。状态字段现有长度已满足，无需数据库迁移。

废弃不删除需求、关联分类记录或 Outline 文档。废弃后仍可查看文档，但后端和前端都禁止编辑、AI 分类、人工确认、功能覆盖修改、查重和合并。列表显示“已废弃”，看板增加“已废弃”列。

## 接口与错误处理

- `POST /api/v1/requirements`：AI 成功且 Outline 写入成功后返回需求；失败沿用 `AI_*`、`AI_NOT_CONFIGURED`、`OUTLINE_*` 或 `CONFLICT`。
- `PUT /api/v1/requirements/{id}`：沿用版本冲突；重生成时 Outline 修订冲突返回 `CONFLICT`。
- `POST /api/v1/requirements/{id}/abandon`：重复废弃、已合并或版本不一致返回 `CONFLICT`。
- 所有写接口继续由控制器先执行项目访问校验。

## 测试

- 后端单元/集成测试验证 AI 接收模版与上下文、AI 输出写入 Outline、无效 AI 输出回滚、编辑两种模式、旧需求补建文档、废弃乐观锁与终态操作限制。
- 前端测试验证编辑字段回填、两种保存请求、覆盖提示、废弃确认、中文状态、废弃后动作禁用和看板列。
- 完成后运行后端测试、前端测试与构建，再合并到本地 `main` 并重建本地服务供人工验收。
