## 1. 目标

参考 Rainier 的产品、产品模块和产品功能管理思路，在现有系统中建设一级业务模块“产品中心”。它统一管理产品主数据、三级模块树、功能库和版本功能清单，并打通项目、需求与标准化中心。

本次采用“在现有系统原生扩展”的方案，不复制 Rainier 页面或数据，不建立双写和同步通道。Rainier 只是功能语义参考，实现继续使用本工程的 React + Java 8 + MySQL 模块化单体架构和飞书项目风格。

## 2. 范围

### 2.1 本次交付

- 一级导航“产品中心”与产品列表/卡片视图。
- 产品详情、生命周期、负责人和基本信息管理。
- 最多三级的产品模块树。
- 挂在末级或非末级模块下的产品功能库。
- 产品版本与版本功能清单。
- 需求与多个产品功能的覆盖关系。
- 未覆盖需求进入标准化候选，并可转换为产品功能。
- 与现有项目版本绑定、标品能力卡、知识条目和审计的兼容。

### 2.2 本次不做

- 不向 Rainier 同步数据，不提供导入器或定时同步任务。
- 不做需求、产品功能和版本之间的自动 AI 关联。
- 不做版本分支、版本差异合并和图形化路线图。
- 不做通用自定义属性引擎。
- 不引入新的前端状态框架、工作流引擎或微服务。

## 3. 信息架构与页面

主导航在“项目空间”和“需求工坊”之间新增“产品中心”，路径为 `/products`。权限为 `product:read`。

### 3.1 产品列表 `/products`

- 默认使用高密度表格，保留卡片视图切换。
- 支持按关键词、分类、状态和负责人筛选。
- 展示编码、名称、分类、负责人、生命周期、模块数、功能数、最新版本和更新时间。
- 新建和编辑使用抽屉，表单保持与当前 Ant Design 交互一致。

### 3.2 产品详情 `/products/:productId`

页头展示产品名称、编码、状态、负责人和快捷操作，下方包含四个页签：

1. **概览**：基本信息、模块/功能/版本统计、最近更改。
2. **模块与功能**：左侧模块树，右侧当前模块的功能列表。选择树节点后只更新右侧内容。
3. **版本**：版本列表和可搜索的版本功能清单。
4. **需求覆盖**：按功能查看已关联需求、完全/部分覆盖数和未覆盖候选。

树节点新增、重命名、移动、废弃与功能新增/编辑使用紧凑抽屉或确认弹窗。加载、空数据、错误、无权限和已归档只读状态均有明确页面反馈。

### 3.3 系统管理入口

原“系统管理 > 产品目录”不再保留第二套编辑页面。`/admin/products` 跳转到 `/products`，从而只保留一个产品事实源和一套管理交互。

## 4. 领域模型

### 4.1 产品与版本

`product` 扩展：

- 新增 `organization_id`、`description`。
- 编码在组织内唯一。
- 生命周期为 `PLANNING` → `ACTIVE` → `SUNSET` → `ARCHIVED`。
- 保留 `owner_user_id`、`category`、`version` 乐观锁和审计时间。

`product_version` 扩展：

- 版本状态为 `PLANNING`、`RELEASED`、`SUNSET`、`ARCHIVED`。
- 版本名在产品内唯一。
- `release_date` 在规划阶段可为空；发布时必须填写。

### 4.2 模块与功能

`product_module`：

- 字段：`product_id`、`parent_id`、`code`、`name`、`description`、`status`、`owner_user_id`、`sort_order`、`version` 和审计时间。
- 状态：`PLANNING`、`ACTIVE`、`DEPRECATED`。
- 编码在产品内唯一，父模块必须属于同一产品。
- 模块树最多三级，移动时禁止成环、跨产品和超深度。

`product_feature`：

- 字段：`product_id`、`module_id`、`code`、`name`、`description`、`status`、`owner_user_id`、`version` 和审计时间。
- 状态：`PLANNING`、`ACTIVE`、`DEPRECATED`。
- 编码在产品内唯一；所属模块必须属于同一产品。
- 功能是产品主数据，不直接从属于某个版本。

`product_version_feature`：

- 建立版本与功能的多对多关系。
- 在版本内的状态为 `INCLUDED`、`PLANNED`、`REMOVED`。
- 一个功能可在多个版本出现，但版本和功能必须属于同一产品。

### 4.3 需求覆盖与标准化追溯

`requirement_product_feature` 建立需求与产品功能的多对多关系，其中 `coverage_type` 为 `FULL` 或 `PARTIAL`，同时记录关联人、关联时间和来源。只允许关联需求所属项目已绑定产品下的功能。

标准化候选继续使用现有 `standardization_debt`：

- 新增 `standardization_debt_requirement` 保留候选与来源需求的多对多关系。
- `standardization_debt` 新增可空 `converted_feature_id`，用于记录候选最终转换的产品功能。
- 转换时用户明确选择目标产品、模块和是否加入某个规划版本，系统不自动猜测归属。

## 5. 核心业务规则

### 5.1 生命周期和引用规则

| 状态 | 可编辑 | 可供新项目绑定 | 历史项目继续使用 |
|------|--------|------------------|----------------------|
| 规划中 | 是 | 否 | 不适用 |
| 生效/已发布 | 是 | 是 | 是 |
| 退市中 | 是，但显示警告 | 否 | 是 |
| 已归档 | 否，只读 | 否 | 是 |

产品、版本、模块或功能一旦被业务数据引用，不提供物理删除，只能退市、废弃或归档。未被引用的草稿数据也优先归档，不在首期 UI 提供删除入口。

### 5.2 版本功能清单

1. 产品经理创建规划版本。
2. 从当前产品功能库选择功能，加入版本清单。
3. 发布前可将功能标记为“已包含”或“规划中”；从历史版本退出的功能标记为“已移除”以保留轨迹。
4. 版本转为 `RELEASED` 时必须有发布日期，且至少包含一个 `INCLUDED` 功能。

### 5.3 项目绑定

- 项目仍同时保存 `product_id` 和 `product_version_id`。
- 新建或更换绑定时，只显示 `ACTIVE` 产品的 `RELEASED` 版本。
- 版本必须属于所选产品；后端重新校验，不信任前端级联结果。
- 历史项目的绑定不因产品退市或归档而被改写。

### 5.4 需求覆盖与转产品功能

1. 需求详情中可搜索并关联多个产品功能，每条关系独立标记完全或部分覆盖。
2. 已有任一 `FULL` 关联时，需求不进入未覆盖列表；只有 `PARTIAL` 或无关联时，可人工创建标准化候选。
3. 候选的“转为产品功能”操作打开表单，预填标题和描述，用户选择目标模块、功能编码、负责人和可选规划版本。
4. 单个事务创建产品功能、版本清单关系（如已选版本）、需求覆盖关系，并将标准化债务更新为 `INCLUDED`。
5. 任一子操作失败时整体回滚，不留下半完成的追溯数据。

## 6. API 与模块边界

产品能力继续归属后端 `catalog` 包，不另建微服务。读取 API 要求 `product:read`，写入 API 要求 `product:write`。

| 用途 | API |
|------|-----|
| 产品列表/详情 | `GET /api/v1/products` / `GET /api/v1/products/{id}` |
| 产品新建/修改 | `POST /api/v1/products` / `PUT /api/v1/products/{id}` |
| 模块树 | `GET/POST /api/v1/products/{id}/modules` |
| 模块修改/移动 | `PUT /api/v1/products/{id}/modules/{moduleId}` |
| 功能列表/新建 | `GET/POST /api/v1/products/{id}/features` |
| 功能修改 | `PUT /api/v1/products/{id}/features/{featureId}` |
| 版本列表/新建 | `GET/POST /api/v1/products/{id}/versions` |
| 版本修改 | `PUT /api/v1/products/{id}/versions/{versionId}` |
| 版本功能清单 | `GET/PUT /api/v1/products/{id}/versions/{versionId}/features` |
| 需求功能关联 | `GET/PUT /api/v1/requirements/{requirementId}/product-features` |
| 候选转功能 | `POST /api/v1/standardization/debts/{debtId}/convert-to-feature` |

项目、需求和标准化模块只能通过 `catalog` 应用服务校验产品归属和状态，不在各自服务中重复实现规则。

## 7. 权限与数据范围

- 新增 `product:read` 和 `product:write` 权限。
- 系统管理员和产品经理拥有读写权限；PMO、交付负责人、交付工程师和技术经理拥有读权限，以保证项目、需求、标准化和知识中的选择器可用。
- 所有查询和写入都以当前用户 `organization_id` 为边界；跨组织 ID 按资源不存在处理。
- 产品负责人、模块负责人和功能负责人必须属于同一组织。
- 所有写操作记录资源类型、资源 ID、操作、操作人、时间和可读摘要。

## 8. 一致性、校验和错误处理

- 所有可编辑主体使用 `version` 乐观锁。版本不匹配返回 409，前端保留用户输入并提示重新加载。
- 编码重复、版本名重复、模块超过三级、树成环、跨产品关联和违法状态迁移返回可读业务错误。
- 不存在或跨组织资源返回 404，无操作权限返回 403，乐观锁或唯一冲突返回 409，字段校验返回 400。
- 归档、退市和废弃不清除历史关联；读取历史项目和需求时仍可显示名称及状态。
- 不使用 Redis 保存产品主数据。首期直接查 MySQL；如未来增加查询缓存，必须在同一服务的写入成功后失效。

## 9. 数据迁移与兼容

新建单个 Flyway 迁移，顺序先加可空字段、回填、再增加非空与外键/唯一约束：

1. 为现有 `product` 回填默认组织，将唯一约束从全局 `code` 改为 `(organization_id, code)`。
2. 现有产品 `ACTIVE` 映射为 `ACTIVE`，`DISABLED` 映射为 `ARCHIVED`。
3. 现有版本 `ACTIVE` 映射为 `RELEASED`，`DISABLED` 映射为 `ARCHIVED`。
4. 新建模块、功能、版本功能、需求功能和标准化追溯表。
5. 不修改 `delivery_project.product_id/product_version_id`、`product_baseline.product_version_id`、`knowledge_item` 的现有外键值。

迁移后必须验证：产品和版本数量不变，项目绑定不变，标准化和知识关联不变，且所有旧数据都能通过新 API 读取。

## 10. 测试与验收

### 10.1 后端

- 迁移集成测试覆盖旧状态映射、组织回填和外键保留。
- 服务测试覆盖三级树、成环阻止、跨产品阻止、状态迁移、版本清单和引用后不可删除。
- 接口测试覆盖 `product:read/write`、组织隔离、403/404/409 语义和审计记录。
- 事务测试覆盖标准化候选转功能成功和中途失败回滚。

### 10.2 前端

- 组件测试覆盖产品表格/卡片切换、筛选、模块树选择、四种页签和版本功能编辑。
- 验证无权限、空数据、请求失败、归档只读和乐观锁冲突反馈。
- 验证原系统管理产品目录跳转且不产生循环。

### 10.3 端到端主链路

1. 产品经理创建产品，建立三级模块树和产品功能。
2. 创建版本，维护功能清单并发布。
3. 交付负责人创建项目并绑定该版本。
4. 录入需求，关联多个产品功能并标记部分覆盖。
5. 创建标准化候选，将其转为新产品功能并加入规划版本。
6. 查看从产品功能回到标准化候选和原需求的完整追溯。

## 11. 验收标准

- 产品中心作为一级模块可独立使用，且系统管理不再产生重复维护入口。
- 产品、最多三级模块树、功能库、版本及功能清单可完整维护。
- 新项目只能绑定生效产品的已发布版本，旧项目关联完整保留。
- 一个需求可关联多个产品功能，并分别表达完全覆盖和部分覆盖。
- 未完全覆盖需求可进入标准化候选，候选可在事务内转为产品功能并保留追溯。
- 组织隔离、权限、审计、乐观锁和所有跨主体归属校验在后端强制执行。
- 现有项目、标准化和知识数据通过迁移后回归测试。
