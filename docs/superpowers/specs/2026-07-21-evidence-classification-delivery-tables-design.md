# 基于证据的需求分类与交付表设计

## 背景

当前需求分类只把需求标题和描述发送给大模型，返回 `level`、`confidence` 和 `reason`。它没有读取已生成的需求调研报告、产品版本功能目录、功能设计 Spec 或人工维护的功能覆盖关系，因此只能作为粗略建议，也不能形成后续建设和投产依据。

本次将分类升级为“完整目录筛选 + 相关资料分析”的两阶段流程，并在每次 AI 分类建议中同时产出建设内容表和投产计划表。AI 建议仍需人工确认，只有人工结论进入漏斗。

## 方案比较

### 方案 A：单次全量分析

一次请求发送需求报告、全部功能和全部 Spec。实现短，但产品功能较多时容易超过模型上下文或超时，且无关 Spec 会降低判断质量。

### 方案 B：两阶段相关功能分析（采用）

第一次请求发送完整的紧凑功能目录，选择最多 12 个相关功能；人工已关联的功能强制纳入候选。第二次请求再发送需求报告、项目计划日期、完整功能目录摘要、人工覆盖关系及候选功能 Spec，输出分类和两张表。

该方案确保全部功能都参与候选判断，同时只加载相关 Spec，兼顾完整性、成本和稳定性。

### 方案 C：向量检索/RAG

为需求和 Spec 建立向量索引后检索。长期扩展性最好，但会增加切片、索引重建、向量存储和运维，本次不引入。

## 数据来源

分类必须使用当前组织和当前需求可访问的数据：

1. 需求结构化字段：编号、标题、描述、来源、优先级。
2. Outline 中的正式需求调研报告正文。
3. 项目、客户、产品和产品版本，以及项目开始日期和计划结束日期。
4. 产品版本完整功能目录：模块、功能编码、名称、描述、版本可用性。
5. 人工维护的需求功能覆盖：`FULL` 或 `PARTIAL`。
6. 候选功能关联的 Outline 功能设计 Spec。

需求没有正式调研报告时拒绝智能分类并提示先生成报告。功能 Spec 缺失不阻断，但必须出现在资料缺口中，相关结论不得伪造。

## 两阶段 AI 流程

### 第一阶段：候选功能筛选

输入需求报告与产品版本完整紧凑功能目录，输出唯一字段 `featureIds`。系统只接受属于当前产品目录的整数 ID，去重并限制 AI 候选最多 12 个；人工覆盖功能不受此上限影响并强制加入。

### 第二阶段：证据分类和表格生成

输入第一阶段候选、相关 Spec、完整功能目录摘要、人工覆盖以及项目上下文。返回严格 JSON：

- `level`: `L0 | L1 | L2`
- `confidence`: `0..1`
- `reason`: 分类摘要
- `evidence`: 可追溯依据列表
- `warnings`: 缺失或待确认资料列表
- `constructionContents`: 建设内容行
- `productionPlan`: 投产计划行

两张表都至少包含一行。L0 可用“配置/验证”说明无代码改动；L2 用“范围外/待立项”明确当前产品不建设。事实不足时字段填“待确认”。

## 建设内容表

每行字段如下：

| 字段 | 含义 |
|---|---|
| `moduleName` | 所属模块；新增能力填建议模块 |
| `featureCode` | 现有功能编码；新增能力填 `NEW` |
| `featureName` | 功能名称 |
| `versionAvailability` | 当前版本可用性或 `待确认` |
| `currentCapability` | 基于功能目录和 Spec 的当前能力 |
| `gap` | 与需求报告的具体差距 |
| `changeType` | `CONFIGURATION`、`ENHANCEMENT`、`NEW_FEATURE`、`INTEGRATION`、`DATA`、`NON_FUNCTIONAL` 或 `OUT_OF_SCOPE` |
| `constructionContent` | 可执行的建设内容 |
| `acceptanceCriteria` | 可验证的验收标准 |
| `priority` | `P0 | P1 | P2 | P3` |
| `evidence` | 对应需求报告章节或功能 Spec 依据 |

## 投产计划表

每行字段如下：

| 字段 | 含义 |
|---|---|
| `phase` | 分析、设计、开发、联调、测试、投产准备、上线或观察等阶段 |
| `workItem` | 本阶段工作内容 |
| `ownerRole` | 产品、研发、测试、实施、客户等责任角色 |
| `plannedStart` | `YYYY-MM-DD` 或 `待确认` |
| `plannedEnd` | `YYYY-MM-DD` 或 `待确认` |
| `deliverable` | 阶段交付物 |
| `entryCriteria` | 准入条件 |
| `exitCriteria` | 准出条件 |
| `riskAndRollback` | 风险、回退或上线撤回策略 |

## 存储与兼容

Flyway 新增 `classification_suggestion.details_json LONGTEXT NULL`，保存第二阶段完整 JSON。现有 `suggested_level`、`confidence` 和 `reason` 继续作为可查询摘要。历史建议的 `details_json` 为空，前端保持原展示，不做破坏性迁移。

需求列表和详情读取最新一条建议，并解析为顶层字段 `classificationEvidence`、`classificationWarnings`、`constructionContents` 和 `productionPlan`。重复请求分类会新增建议记录，历史数据保留，当前界面展示最新记录。

## 前端交互

分类抽屉加宽。AI 建议区展示分类、置信度、原因、证据和资料缺口；下方以“建设内容表”和“投产计划表”两个中文页签展示完整表格。人工可沿用建议或改判；改判仍必须填写原因。AI 服务不可用时保留纯人工确认能力。

## 错误与边界

- 无正式需求报告：返回明确冲突错误，不生成低质量建议。
- AI 候选包含越权或不存在功能：丢弃非法 ID；人工覆盖功能仍保留。
- AI 最终结果字段缺失、类型错误、枚举非法、表格为空或单元格空白：按不兼容响应失败，不写入建议。
- Outline 中某个候选 Spec 无法读取：记录资料缺口并继续，提示人工复核。
- AI 调用失败：事务不写入 `classification_suggestion`，原需求和既有建议不变。

## 测试

- 迁移测试验证 `details_json` 存在。
- 后端服务测试验证两阶段调用、全部功能目录、需求报告、覆盖关系、相关 Spec、项目日期和严格响应校验。
- API 测试验证两张表持久化和读取。
- 前端测试验证两张表、证据、资料缺口、人工确认与改判交互。
- 最终运行后端全量测试、前端全量测试和生产构建。

## 明确不做

- 不引入向量数据库或 Embedding 服务。
- 不自动修改人工功能覆盖关系。
- 不自动确认 AI 分类。
- 不为两张表额外创建 Outline 文档或导出接口；结构化结果先作为分类建议的一部分保存和展示。
