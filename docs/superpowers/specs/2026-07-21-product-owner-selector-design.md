# 产品中心负责人选择设计

## 背景

产品中心目前在产品、模块和功能编辑表单中直接填写用户 ID，列表和详情也以 `#ID` 展示负责人。该方式容易填错，并且无法保证所选人员确实承担产品管理职责。

系统现有内置角色为“产品经理”，角色编码是 `PRODUCT_MANAGER`。本设计将该角色作为产品负责人候选来源，不新增角色、不调整权限模型。

## 目标

- 产品、模块和功能的负责人统一从当前组织内具有 `PRODUCT_MANAGER` 角色的启用用户中选择。
- 前端显示负责人姓名，不再要求用户识别或输入用户 ID。
- 后端校验负责人资格，避免绕过前端提交任意用户 ID。
- 保持现有 `ownerUserId` 数据字段及数据库结构不变。

## 非目标

- 不新增 `PRODUCT_OWNER` 角色。
- 不修改产品经理角色现有权限。
- 不改造产品中心以外的负责人字段。
- 不引入新的通用人员选择组件或第三方依赖。

## 方案

### 候选人员接口

在产品中心 API 下增加只读接口：

`GET /api/v1/products/owner-options`

接口仅返回满足以下条件的用户：

1. 与当前登录用户属于同一组织；
2. 用户状态为 `ACTIVE`；
3. 通过 `user_role` 关联到角色编码 `PRODUCT_MANAGER`。

响应只包含选择器所需字段：

```json
[
  {
    "id": 21,
    "displayName": "张产品"
  }
]
```

候选项按显示名称、用户 ID 稳定排序。没有符合条件的人员时返回空数组。

### 后端校验

产品、模块和功能保存时，如果提交了 `ownerUserId`，后端必须用与候选接口相同的组织、状态和角色条件校验。校验失败返回参数错误，并提示“请选择当前组织内启用的产品负责人”。未选择负责人仍然允许保存。

这样即使调用方绕过前端，也不能把普通用户、停用用户或其他组织用户设置为负责人。

### 姓名展示

产品、模块和功能查询结果增加可选字段 `ownerName`。查询通过用户表读取显示名称，使已有负责人即使后来停用或被移除角色，历史数据仍能展示姓名。

候选资格与历史展示分离：候选接口只返回当前有效人选，数据查询仍展示记录上已经关联的负责人。

### 前端交互

产品中心进入页面后按需加载负责人候选项，并在以下位置复用同一份数据：

- 产品新建和编辑抽屉；
- 模块新建和编辑表单；
- 功能新建和编辑表单；
- 产品列表负责人筛选。

负责人字段使用 Ant Design `Select`：

- 标签统一为“负责人”；
- 支持按姓名搜索；
- 支持清空，负责人保持选填；
- 加载期间显示加载状态；
- 空候选时显示“暂无产品负责人，请先在系统管理中配置产品经理角色”；
- 列表、详情和筛选优先显示 `ownerName`，缺失时显示“未指定”，不向用户展示原始 ID。

只读或归档状态沿用现有禁用规则。

## 数据流

1. 前端请求产品数据和负责人候选项。
2. 后端按当前组织及 `PRODUCT_MANAGER` 角色返回有效候选人。
3. 用户通过姓名选择负责人，前端仍提交现有的 `ownerUserId`。
4. 后端再次校验该 ID 的组织、状态和角色资格后保存。
5. 查询接口返回 `ownerUserId` 和 `ownerName`，前端只展示姓名。

## 兼容性与错误处理

- 不修改表结构，现有记录无需迁移。
- API 仅新增字段和端点，不删除现有字段。
- 已有关联负责人不再符合候选资格时，其姓名仍可查看，但重新保存负责人时必须选择当前有效人选或清空。
- 候选接口失败时，表单保留其他字段可见，并显示现有请求错误；不退化为手填 ID。

## 测试范围

### 后端

- 候选接口只返回当前组织内、启用且具有 `PRODUCT_MANAGER` 角色的用户。
- 普通用户、停用用户和其他组织用户不会进入候选列表。
- 产品、模块和功能拒绝不符合资格的负责人。
- 产品及结构查询返回负责人姓名。

### 前端

- 产品编辑器显示姓名选择器且不再出现负责人 ID 输入框。
- 模块和功能编辑器使用同一候选列表。
- 选择负责人后保存请求仍正确提交 `ownerUserId`。
- 产品列表、详情及负责人筛选显示姓名。
- 无候选人员时显示明确的空状态提示。
