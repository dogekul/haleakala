# 需求采集生成 Outline 文档设计

## 背景

需求工坊目前在“采集需求”抽屉中填写项目、标题、描述、来源和优先级，提交后只创建 MySQL 需求记录。项目已经将 Outline 作为文档中心，也已经通过知识库维护唯一启用且已发布的“需求调研报告模版”，但需求记录尚未关联文档。

## 目标

- 用户填写完需求采集单后，一次提交同时创建需求和 Outline 文档。
- 文档基于当前组织已发布的“需求调研报告模版”生成，不能残留 `{{...}}` 占位符。
- 文档存放在对应项目的“需求文档”目录中。
- 需求记录保存 Outline 关联、来源模版和模版版本，便于追溯。
- 旧需求继续正常展示；没有关联文档时不显示“查看文档”操作。

## 方案

提交仍使用现有 `POST /api/v1/requirements`。后端在一个业务事务内：

1. 校验项目访问权并创建需求记录，生成需求编号。
2. 查找当前组织唯一启用、已发布、场景为 `OPPORTUNITY_RESEARCH` 的文档模版。
3. 使用需求、项目、客户、提交人和当前日期替换模版中的已知字段；在正文顶部写入需求编号、标题、来源、优先级和完整描述；其余无法从采集单获得的占位符填为“待确认”。
4. 确保 Outline 中存在“项目文档 / 当前项目 / 需求文档”目录，创建业务键为 `REQUIREMENT:{id}:RESEARCH_REPORT` 的文档。
5. 将 Outline 链接 ID、来源模版 ID 和来源模版版本写回需求记录。

Outline 或模版处理失败时，请求失败且数据库事务回滚，前端保留采集表单并展示错误。Outline 使用确定性业务键恢复重试，避免重复创建文档。

## 数据模型与接口

`requirement_item` 新增：

- `outline_link_id`：关联 `outline_document_link`。
- `source_template_id`：关联生成时使用的知识库模版。
- `source_template_revision`：生成时使用的已发布版本。

需求列表返回 `outlineLinkId`。新增 `GET /api/v1/requirements/{id}/document`，在验证组织和需求访问权后返回当前 Outline 文档视图及 URL。

## 前端交互

- 抽屉主按钮由“保存草稿”改为“完成采集并生成文档”，明确提交结果。
- 成功提示改为“需求已创建，调研文档已保存到 Outline”。
- 已关联文档的需求在操作区显示“查看文档”，点击后通过后端读取最新文档地址并在新窗口打开。
- 不增加第二套 Markdown 编辑器；正式文档后续在 Outline 中维护。

## 边界与兼容性

- 没有可用模版时阻止创建并给出明确错误，避免生成不可追溯的空文档。
- 旧需求的三个新字段均可为空，无需迁移历史数据。
- 本次不改变需求后续分类、确认和转需求流程。
- 外部 Outline 与本地事务无法做到分布式原子提交；确定性业务键保证失败重试不会生成重复文档。
