# 合规审查系统需求调研报告

> 本报告用于确认甲方核心需求并作为后续方案设计、工作量评估和验收的依据。标注“待确认”的内容需在项目启动前由甲方明确。

## 1. 项目概述

### 1.1 建设背景

甲方当前合规审查主要依赖人工检索、经验判断和线下流转，存在规则分散、标准不一、审查效率低、过程难追溯、整改闭环不完整等问题。拟建设统一的合规审查系统，实现审查规则集中管理、材料自动初审、人工复核、问题整改及全过程留痕。

### 1.2 建设目标

1. 统一维护法律法规、行业规范及内部制度，形成可执行的审查规则库。
2. 对业务材料进行自动识别、规则匹配和风险提示，降低重复人工工作。
3. 建立“提交—初审—复核—整改—归档”的线上闭环。
4. 确保审查结论有依据、过程可追溯、权限可控制、数据可统计。

## 2. 调研范围

本次调研覆盖以下内容：

- 审查对象：合同、制度文件、业务方案、营销材料及其他待确认材料。
- 审查依据：法律法规、监管要求、行业标准、内部制度及历史案例。
- 审查流程：任务发起、材料提交、自动初审、人工复核、整改复审、结果归档。
- 支撑能力：规则管理、权限管理、审计留痕、统计分析及外部系统集成。

不在本期默认范围内：替代法务或合规人员作出最终判断、自动对外出具法律意见、未经授权调用外部敏感数据。

## 3. 用户与职责

| 角色 | 主要职责 |
|---|---|
| 业务提交人 | 发起审查、上传材料、补充说明、落实整改 |
| 合规审查人 | 复核系统提示、提出意见、确认风险等级 |
| 合规负责人 | 分派任务、处理重大风险、审批最终结论 |
| 规则管理员 | 维护规则、适用范围、版本及生效时间 |
| 系统管理员 | 管理组织、用户、角色、参数和接口配置 |
| 审计人员 | 查询审查记录、操作日志和历史版本 |

## 4. 现状与主要问题

根据通用场景形成以下初步判断，需由甲方访谈确认：

- 审查规则散落在文件、邮件或个人经验中，更新后难以及时同步。
- 相似材料被重复审查，常见问题缺少复用机制。
- 审查意见与原文位置、规则依据关联不足，沟通成本较高。
- 线下流转无法准确统计处理时长、逾期情况和风险分布。
- 整改结果缺少复审和关闭记录，责任边界不清晰。
- 敏感材料的访问、下载和导出控制不足。

## 5. 目标业务流程

1. **发起任务**：选择审查类型，填写业务背景、紧急程度和期望完成时间。
2. **提交材料**：上传文件或从业务系统引用材料，系统检查格式和完整性。
3. **自动初审**：解析内容，匹配适用规则，标注风险位置、等级、依据和修改建议。
4. **人工复核**：审查人员确认、调整或驳回系统结论，并补充专业意见。
5. **整改复审**：业务人员逐项反馈并提交新版本，审查人员核验整改结果。
6. **结论归档**：形成审查报告，保留材料版本、处理记录和最终结论。

原则上，系统自动结果仅作为辅助建议，最终结论由甲方授权人员确认。

## 6. 功能需求

优先级说明：P0 为上线必需，P1 为重要增强，P2 为后续优化。

| 模块 | 核心需求 | 优先级 |
|---|---|---|
| 审查任务 | 新建、分派、转办、催办、撤回；支持按类型配置流程 | P0 |
| 材料管理 | 上传、预览、下载、版本对比、附件管理；支持常用办公文档格式 | P0 |
| 内容解析 | 提取正文、表格及关键字段；解析失败时提示并允许人工处理 | P0 |
| 自动审查 | 按材料类型匹配规则，定位问题原文，输出风险等级、依据和建议 | P0 |
| 人工复核 | 对系统结果确认、修改、忽略或补充；忽略时填写理由 | P0 |
| 整改闭环 | 问题指派、整改反馈、复审、关闭和逾期提醒 | P0 |
| 规则库 | 规则新增、审批、发布、停用、版本管理及生效范围配置 | P0 |
| 报告管理 | 生成审查结论及问题清单，支持导出；模板由甲方配置 | P0 |
| 检索统计 | 按任务、部门、风险、规则等检索；统计数量、时效和高频问题 | P1 |
| 案例沉淀 | 将已确认问题和优秀整改结果沉淀为参考案例 | P1 |
| 消息提醒 | 待办、退回、逾期、规则变更等站内或外部消息提醒 | P1 |
| 智能增强 | 相似案例推荐、自然语言问答、规则冲突提示 | P2 |

## 7. 规则库要求

- 每条规则至少包括：名称、来源、适用对象、触发条件、风险等级、审查提示、修改建议、责任部门、生效及失效时间。
- 规则发布前应经过审核；发布、变更和停用均保留版本及操作记录。
- 规则应支持按组织、业务类型、区域和材料类型设置适用范围。
- 审查结论必须能够追溯到具体规则及其当时生效版本。
- 法规原文及授权范围由甲方提供并确认，系统不得使用来源不明的规则作为正式依据。

## 8. 非功能需求

| 类别 | 要求 |
|---|---|
| 安全 | 身份认证、最小权限、敏感字段脱敏、传输与存储加密、下载与导出控制 |
| 审计 | 记录登录、查看、上传、下载、规则变更、结论调整和导出等关键操作 |
| 性能 | 常规页面响应、并发量、单文件大小及审查时长指标由甲方结合业务量确认 |
| 可用性 | 核心服务可用性、备份频率、恢复时间目标和容灾等级待确认 |
| 兼容性 | 浏览器、操作系统、文件格式及移动端范围待确认 |
| 可维护性 | 规则、流程、表单、报告模板和消息策略应尽量支持配置化维护 |
| 可解释性 | 每项风险提示应展示命中位置、规则依据和判断说明，禁止只给出结论 |

## 9. 数据与系统集成

建议优先确认以下接口：

- 统一身份认证及组织用户系统。
- OA/BPM 待办与审批系统。
- 合同、采购、营销或其他业务系统。
- 企业文档库、法规库及消息平台。

接口应明确数据范围、同步方向、调用频率、失败补偿和责任系统。未经甲方批准，审查材料不得用于外部模型训练或跨境传输。

## 10. 验收要点

1. 甲方确认范围内的审查流程可完整线上闭环，状态及责任人清晰。
2. 规则可配置、可审批、可版本追溯，历史任务可还原当时依据。
3. 自动审查结果能够定位原文并说明依据，人工可复核和修正。
4. 同一任务的材料、意见、整改和结论形成完整记录。
5. 权限隔离、敏感操作审计、数据导出控制符合甲方安全要求。
6. 约定的性能、可用性、兼容性和接口联调指标通过测试。
7. 自动审查准确率、召回率或人工采纳率应基于甲方标注样本共同制定，不建议在缺少样本时预设固定指标。

## 11. 待甲方确认事项

| 序号 | 待确认事项 | 对项目的影响 |
|---|---|---|
| 1 | 首期审查对象及业务边界 | 决定功能范围和规则建设量 |
| 2 | 适用法规、内部制度及现有规则来源 | 决定规则库初始数据质量 |
| 3 | 现行流程、审批层级及重大风险升级机制 | 决定流程配置方案 |
| 4 | 日均任务量、并发量、文件类型和文件大小 | 决定性能与资源配置 |
| 5 | 部署方式及是否允许调用外部模型 | 决定技术架构和数据安全方案 |
| 6 | 需对接的系统、接口责任方及联调窗口 | 决定集成计划和工期 |
| 7 | 数据分级、保存期限、备份及销毁要求 | 决定安全和存储方案 |
| 8 | 验收样本、评价方法及目标指标 | 决定测试与验收口径 |

## 12. 调研结论

甲方核心诉求可归纳为：**统一规则、提高审查效率、保留人工决策权、实现整改闭环、确保数据安全和全过程可追溯**。

建议首期以 P0 功能形成最小闭环，并选择 1—2 类高频材料开展试点。待规则、样本和流程稳定后，再扩展智能问答、案例推荐及更多业务系统集成。
